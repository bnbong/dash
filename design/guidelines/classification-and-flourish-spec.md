# Dash — 기록 분류 & 해석 설계 스펙

> **상태:** 초안 v0.1 (2026-07) · 기획서 §6·§7·§16–18 기반 · MVP(Phase 1) 구현 대상
> **목적:** 기록(자연어) → 삶의 영역·활동 유형·행복의 세 축·의미 해석으로 자동 변환하는 파이프라인의 규칙을 정의한다. 기획서에 "자동 분류(P0)"로 명시돼 있으나 *어떻게* 분류하는지는 미정이었다. 이 문서가 그 공백을 메운다.

---

## 0. 파이프라인 개요

한 건의 기록은 아래 단계를 거친다.

```text
raw_input (자연어 한 줄 / 음성 STT / 사진 캡션)
  → [1] 정규화·전처리 (길이·언어·비어있음 체크)
  → [2] 분류·해석 (LLM 1회 호출, 구조화 JSON 출력)
  → [3] 후처리 (신뢰도 게이팅·Seed 자격 판정)
  → [4] 사용자 확인·수정 (RecordReview UI)
  → activity 레코드 확정 → Dashboard·Evidence·Grove 반영
```

**설계 원칙:** 분류는 규칙(키워드 매칭)이 아니라 **맥락 이해(LLM)** 로 한다. "밀린 공과금 냈다"는 키워드에 '공과금'이 없어도 `Life Admin / 유지`로 이해돼야 한다. 규칙 매칭은 오프라인 폴백(§4)에서만 쓴다.

**출력 데이터(기획서 §17.1과 정합):** `domain(주)`, `secondary_domains[]`, `activity_type`, `axes[]`, `energy_level`, `emotional_context`, `interpreted_meaning`, `evidence_sentence`, `impact_seed_eligible`, `confidence`.

## 공백 1 — 다중 영역 분류 (Primary + Secondary)

**문제:** "밴드 합주 2시간"은 Hobby이자 Relation이다. 단일 영역만 붙이면 의미가 깎인다.

**규칙:**
1. LLM은 항상 **주 영역(primary) 1개**를 정한다 — 사용자가 그 행동으로 "가장 크게 운영한" 삶의 영역. 대시보드 집계·색상·Seed 귀속의 기준이다.
2. **부 영역(secondary)은 최대 2개**까지 허용한다. 명확히 걸칠 때만 부여하고, 애매하면 비운다(과분류 금지).
3. 부 영역은 대시보드 활동량에 **0.5 가중치**로 반영한다(주=1.0). 이렇게 해야 8영역 합계가 왜곡되지 않는다.
4. Impact Seed는 **주 영역에만** 귀속한다(중복 적립 방지).
5. UI 표기: 주 영역은 진한 색 칩, 부 영역은 작은 점/옅은 칩. Evidence·Today 로그 모두 동일 규칙.

**예시**

| 기록 | 주 | 부 | 활동유형 |
|---|---|---|---|
| 밴드 합주 2시간 | Hobby | Relation | 성장 |
| 친구와 등산 | Body | Relation | 회복 |
| 밀린 공과금 납부 | Life Admin | — | 유지 |
| 이력서 문단 수정 | Work & Learning | — | 성장 |
| 어머니와 통화 40분 | Relation | Mind | 유지 |

## 공백 2 — 분류 수정 UX & 학습

**문제:** LLM 분류가 틀릴 수 있고, 사용자는 "내 삶"에 대한 최종 권위자다. 수정 경로가 없으면 신뢰가 깨진다.

**규칙:**
1. 기록 직후 **RecordReview**(확인 카드)를 항상 노출한다 — "이 기록을 이렇게 이해했어요". 주 영역·부 영역·세 축·활동유형을 **탭 한 번으로 수정**할 수 있다.
2. 확인은 **낮은 마찰**이어야 한다: 기본값이 맞으면 그냥 "맞아요"로 넘어가고, 저에너지 모드(기획서 §16)에선 확인 단계를 건너뛰고 조용히 저장한다.
3. **수정 시 톤:** 실수를 사과하지 않는다("잘못 분류했어요" ✗). "이 행동은 당신에게 어떤 영역에 가까운가요?"처럼 **사용자에게 권위를 넘기는** 언어를 쓴다.
4. **경량 학습(Phase 1):** 사용자가 특정 표현을 반복 재분류하면(예: '합주'→항상 Relation 추가) 해당 사용자 프로필에 **few-shot 힌트**로 누적해 다음 프롬프트에 주입한다. 전역 모델 재학습은 Phase 2+.
5. 원문(`raw_input`)은 절대 자동 수정하지 않는다 — 해석만 조정한다.

## 공백 3 — 행복의 세 축 매핑 (성장·즐거움·의미)

**문제:** 세 축은 8영역과 **직교하는 두 번째 렌즈**다(§README 참조). 어떻게 도출하나?

**규칙:**
1. 세 축은 영역에서 기계적으로 파생하지 **않는다**. 같은 "운동"도 성장(기록 경신)일 수도, 즐거움(그냥 상쾌해서)일 수도 있다. → **LLM이 맥락으로 axes[]를 함께 추론**한다.
2. 한 기록은 **1~2개 축**을 가진다(0개 가능: 순수 유지·행정 행동은 세 축 어디에도 강하게 안 붙을 수 있음 — 정상).
3. **판정 가이드(LLM 프롬프트에 주입):**
   - **성장 Growth** — 더 유능해짐·배움·도전·역량 확장. (예: 이력서 수정, 새 레시피, 운동 기록 경신)
   - **즐거움 Enjoyment** — 긍정 정서·놀이·음미·**쉼**. 결과가 아니라 경험 자체가 목적. (예: 밴드 합주, 산책, 좋아하는 드라마, 낮잠)
   - **의미 Meaning** — 연결·기여·돌봄·목적. 나를 넘어선 것. (예: 친구 챙김, 봉사, 가족 통화, Impact Seed 기여)
4. **집계:** 기간별 FlourishBalance는 축별 **기록 수의 비율**로 계산한다(가중치 없음). 빈 축은 점수가 아니라 **부드러운 제안**으로 노출한다("이번 주 즐거움이 조용했어요 — 쉼도 번영입니다"). 최대화해야 할 숫자가 아니다.
5. **직장인 힐링 훅:** 즐거움 축이 2주 연속 최저이고 성장/의미가 높으면 → 번아웃 신호로 보고 허락형 넛지를 띄운다. (경쟁·죄책감 금지)

**영역 × 세 축 경향 (강제 아님, LLM 참고용 사전 확률)**

| 영역 | 자주 겹치는 축 |
|---|---|
| Work & Learning | 성장 |
| Hobby & Creation | 즐거움 · 성장 |
| Relation | 의미 |
| Body | 성장(운동 목표) · 즐거움(산책) |
| Mind | 즐거움(쉼) · 의미 |
| Impact | 의미 |
| Space · Life Admin | (약함 — 대개 유지) |

## 공백 4 — 오프라인 · 비용 · 지연

**문제:** 매 기록마다 LLM 호출은 비용·지연·오프라인 문제를 만든다. 그런데 자동 분류는 P0다.

**규칙 (3-tier):**
1. **즉시 낙관적 표시(온디바이스):** 기록 즉시 가벼운 키워드/사전 휴리스틱으로 **잠정 영역**을 추정해 0ms로 보여준다. "임시" 상태 표시(옅은 점선 칩).
2. **정식 분류(서버 LLM, 비동기):** 백그라운드에서 LLM 호출 → 결과 도착 시 잠정값을 부드럽게 교체. 도착 전에도 로그 저장은 됨(낙관적 UI).
3. **오프라인·실패 폴백:** 네트워크 없으면 온디바이스 추정값으로 확정하고 `pending_reclassify` 플래그. 다음 온라인 시 재분류 배치 실행.

**비용 최적화:**
- **배치:** 저에너지 모드·다건 입력은 모아서 1회 호출(여러 기록을 한 프롬프트에).
- **캐시:** 동일/유사 문장은 사용자 프로필 캐시에서 즉시 매핑.
- **모델 계층:** 기본은 저가·저지연 모델. Dash Plus의 "고급 AI 해석"(§19)만 상위 모델·긴 컨텍스트 사용.
- **해석 문장 생성 분리:** 영역/축 분류(구조화, 저렴)와 긴 해석 문장 생성(비쌈)을 분리 호출 가능. 무료 티어는 짧은 템플릿 해석, Plus는 생성형.

## LLM 인터페이스 스펙

**System prompt (요지):** "너는 Dash의 생활 운영 해석기다. 사용자의 한 줄 기록을 읽고 삶의 영역·활동유형·행복의 세 축·의미를 판정한다. 평가하지 말고, 과장하지 말고, 낮은 에너지를 실패로 보지 마라. 확신이 없으면 부 영역·축을 비워라." (기획서 §18.2 출력 원칙 전면 적용)

**Output schema (JSON):**
```json
{
  "domain": "admin",
  "secondary_domains": [],
  "activity_type": "유지",
  "axes": ["의미"],
  "energy_level": "low",
  "emotional_context": "안정",
  "interpreted_meaning": "생활 마찰을 줄이는 운영 행동",
  "evidence_sentence": "밀린 공과금을 정리해 다음 달의 부담을 덜었다",
  "impact_seed_eligible": true,
  "confidence": 0.82
}
```

**신뢰도 게이팅:** `confidence < 0.55`면 주 영역을 확정하지 않고 RecordReview에서 **사용자에게 먼저 묻는다**("이 기록은 어떤 영역에 가까운가요?"). Seed 적립은 확정 후에만.

**허용 값**
- `domain` / `secondary_domains`: body · space · relation · work · hobby · mind · admin · impact
- `activity_type`: 성장 · 유지 · 회복 · 관계 · 행정 (기획서 §17.1)
- `axes`: growth · enjoyment · meaning (0~2개)
- `energy_level`: low · mid · high · null
- `emotional_context`: 자유 텍스트(피곤·안정·불안·만족 등)

## 디자인 반영 (이 스펙의 UI 구현)

- **RecordReview** 컴포넌트 — 기록 직후 분류 결과를 보여주고 주/부 영역·세 축을 탭으로 수정 + 확인. (`components/dash/`)
- **Today 로그·Evidence** — 주 영역(진한 칩) + 부 영역(옅은 점) 표기 규칙 반영.
- **InterpretationCard** — `axes` prop으로 세 축 태그 노출(이미 구현).
- **저에너지 모드** — RecordReview 건너뛰고 조용히 저장(프로필 토글과 연동).

## 미결·후속 (Phase 2+)

- 전역 모델 파인튜닝 / 재학습 루프
- 사진→행동 추론(비전 모델) 정확도 기준
- 음성 STT 오인식 교정 UX
- 세 축 장기 추세 기반 코칭(직장인 번아웃 예측)
- 다국어(영어 UI) 시 분류 프롬프트 현지화
